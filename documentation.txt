###FAT Library

Now uses the fat library v2.0 in the include directory, this is principally the
same library as was used before, but with the each function in a dedicated file.

###RTOS

Now uses rtos-v3-0 in the include directory, this is an extended version of the
rtos-v2-1 and rtos-v2-2 used before, which includes INTSAVE, QUEUE primitives,
TESOUT etc.

During RVL12 testing I have seen the same issue as with MQTT that suddenly the
rtos no longer serves one of it's jobs. This was clearly related to the way the
RTOS interrupt was acknowledged. v3-0 has a slightly different acknowledge of the
RTOS interrupt and more important we don't have any sbic b_RTOS loops which were
the place where a job hung before. We assume now that the interrupt is started
during the instructions which save the registers and therefore never wait for the
ISR to set b_RTOS again as we can assume that when the ISR "returns" (ret and not reti)
it has set b_RTOS and acknowledged the interrupt.

###MALLOC FREE

malloc() and free() are not related to rtos-v3-0, they no longer require a dedicated
interrupt pin. The way level0 is entered using b_RTOS is now completely generic and
every subroutine can actually enter level0 by just clearing b_RTOS, of course such
a subroutine must use reti and no longer ret in order to exit level0 interrupt state

###CLI

The CLI provides a number of commands. Command Keywords may be abbreviated. Some
commands are primarily used for debugging and are marked as such and are not useful
for normal operation

Numbers can be entered as octal, hex or decimal using the standard linux notation
(e.g. preced octal numbers with a 0 and hexadecimal numbers with 0x)

CLI allows editing of command line and saves the last executed commend, which can 
be recalled with ARROW-UP and reissued or edited before reissuning, handy when
you have a syntax error.

####[RLV12]
This is a dummy command to identify init files on the SD-Card, they must start
with a line that contains only this key, when found the following lines will be
treated as startup commands when a SD-Card is inserted or at power-up. During
startup or insertion the firmware looks for a file named RLV12.INI and then looks
for such a start record. If found it will sequentially treat each line as a 
command and execute it. If no such file is found then it will just look for
linux partitions and then attach the first four partitions found, that match
type and size to the RL units

###Apple II Commands
Built into the CLI is a subset of Apple II monitor commands. This is mostly used
during debugging and usage may corrupt the data of your microcontroller. Use with
care. Currently RAM at 0x5000..0x5FFFF is not used by the firmware and therefore
it should be the only memory you use to modify with these commands. For the syntax
consult the Apple II Monitor Documentation. Apple II commands entered via the CLI
must be preceded with the $ sign.

$5000			Displays the content of memory locaiton 0x5000
$			Displays the content of the next memory locations
$5000.50FF		Displays the content of memory from 0x5000 to 0x50FF
$5000<5200.53FFv	Compares Memory from 0x5200 to 0x53FF with memory starting
			at 0x5000 and shows an error if it does not match
$5000<5200.53FFm	copies the memory from 0x5200 to 0x53FF to 0x5000
$5000.51FFp		writes a pattern by incrementing the value starting with 0.
$5000:01 02 03		Store the values 0x01, 0x02, 0x03 to 0x5000, 0x5001, 0x5002

DMATEST also uses the microcontroller memory. E.g "dmatest 0x5000" writes 512. bytes
from the microcontroller memory via DMA to the host memory using the DMA address
previously set with the "dmaaddr" command and then reads the host memory using DMA
back to the microcontroller memory location 0x5200, i.e. the subsequent 512. bytes
in microcontroller memory, now you can use the "v" (verify) command to check if
the values read back are identical

####Autoboot

The controller has an auto-boot feature built in. You only need to enter the
following command in ODT "174414g" which will load the boot block of unit 0 to
address 0 and execute it. I.e. it will typically boot RL0. This is useful in case
your boot ROM has no bootstrap code for DL

####Autostartup

In fact you can also just enter the ODT command "173000g", which will also activate
the autoboot feature. This is handy in case your system does not have a bootrom at
all and in case you configure your CPU card to start execution at 173000 at power-up
this will allow you to automatically boot form RL0 when your system is powered-up.

Note: the controller does not provide a full boot ROM



###VCB

For each FAT-16 or FAT-32 volume we have a Volume Control Block (VCB). The volume
control blocks are queued in the volqueue. The first vcb in the queue is the active
volume.

Even the firmware reads all FAT partitions and creates multiple volume control blocks,
the handling of multiple FAT volumes is not fully implemented and therefore it is
recommended to use SD-Cards with exactly one FAT-16 or FAT-32 partition.

There are two IO Parameter blocks assiciated with the vcb. One is used for directory
IO and the other one is used for FAT IO. Currently each has it's own sector buffer

Then there are a lot of information from the Volume Boot Record, some information
is only kept and not used

part1start	First Sector of the partition 
fat1start	First Sector of the first FAT
bytespsect	Bytes per Sectors, we only support 512byte sectors
reservedsect	Reserved Sectors at the beginning of the partition after VBR
sectbefore	Sectors before this partition on the physical media (==part1start)
sectpart	Sectors in this partition, i.e. the size of the partition in sectors
rootdir		FAT-16: First Sector of the root directory, FAT-32 first Cluster of
		the root directory
sectperfat	Sectors Per FAT (not a FAT partition typically has two or more copies of
		the FAT as this is the most vulnerable informaiton)
datastart	First sector of the data 

DirPointer	During file name lookup this field is set to the address of the
		directory entry in the sector buffer for directory IO if a match
		of the name occured
DirNxtPtr	This is the pointer to the next directory entry. This is used as the
		starting point for ReadDir
		
		Note: with long file name support more then one directory entry is used
		to describe the file. The long file name directory entries preceed the
		real directory entry, long file name direcotry entries store a index 
		of the portion of the name, a start long file name flag and up to 13
		characters of the file name. The real directory entry has encoded
		8.3 file name and all the other information used to describe the file,
		like start cluster, filesize, create date and creation time

DirCluster	this is the start cluster of the working directory all name lookups
		start using this directory, DirCluster is set to 0 if the root
		directory is requested. This field is updated by the 'cd' command and
		used by the Name2DirEntry function
		
UpdatePtr	During a file name search the function uses this pointer to update
		the portion of the filename in the input name with the correct
		case as found in the directory. Updating is done by the MatchFileName
		function when a match has occured

dirsectors	the size of sectors of the root directory for FAT-16 volumes, not
		used and set to 0 by FAT-32 volumes

sectperclst	Then number of sectors that make up a cluster, a cluster is a 
		contiguous range of sectors therefore we can read up to sectperclst
		sectors by just incrementing the sector number of the IO parameter
		block until we read sectperclst sectors when we need to find the
		cluster linked to the current cluster to proceed with reading sectors
		of the file or directory

DirCount	Counter to keep track of the number of directory entries already
		processed by ReadDir. Not that when an overflow to 32 happens this
		requires that we read another sector from the directory and the 
		counter is reset again

Status		House-keeping and tracking flags

Vol__MBR	Set when we could read the MBR of the partition
Vol__VBR	Set when we could read the VBR of the partition
Vol__FAT32	Set if the FAT partition is a FAT-32 partition
Vol__Long	Set when we are collecting the junks of a long file name
Vol__Linked	Set when the directory we currently process is a linked list
		of clusters, in fact this flag is set except when we are reading
		the root directory of a FAT-16 volume

##FAT Library

The following are the FAT Library functions used to access a FAT-16/32 volume

###OpenDir(struct* VolumeControlBlock)

Opens the directory who's start cluster is placed in Vol_DirCluster for sequential
reading of directory entries. When you want to read the root directory you just
set Vol_DirCluster to 0

###ReadDir(struct* VolumeControlBlock)

When a directory has been opened we continuously call ReadDir. ReadDir will return
a pointer to the real directory entry of the next file. If the file has a long
file name the long file name has been constructed to the global buffer LongFileN
and the Vol__Long flag in Vol_Status will be set

###Name2DirEntry(struct* VolumeControlBlock, char* name)

This routine will translate a name to a directory entry. Search will either start
at the directory who's start cluster is stored in Vol_DirCluster or in case the name
has an absolute path start in the root directory. The function will scan the input
path and check that each element exists. In case there is a .. parent directory
in the input path it will continue with the parent directory of the current directory.
At the end Vol_DirPointer will point to the directory entry in the sector buffer
for directory IO, which can either be the entry for a file or a directory. In case
the root directory has been requested Vol_DirPointer is set to 0.

###MatchFileName

This will try to match the name of the current directory entry with the input name
it covers the case of long file names and also the fact that a 8.3 name in the 
directory is stored differently to the 8.3 name given in input names.

###LinkedCluster

Get the cluster linked to the current cluster.

###Cluster2Sector

Translates a cluster to the first sector in the cluster

###CopyName

Copies a file name to the global buffer NameBuffer from an input path. CopyName will
copy the name until it reaches either DELIM, NULL or CR. In case the name is not 
empty the T-Bit will be set, the name copied is NULL terminated, the pointer to 
the input buffer is updated to point past the DELIM, NULL or CR. Copy Name will
also return the terminating character in R24. This routine is internal to the
FAT-library and must not be called outside

###CreatePath

Create Path allows to combine the input file name and the working directory to 
one absolute path. in case the input file name contains .. to select the parent
directory it will remove the top directory name in the resulting path. The
final result will only have file names. Normally this routein is only called
with a valid working directory and a validated input file name

###FindFreeCluster(struct* VolumeControlBlock, uint16_t Clusters)

not implemented

Tries to find the number of free clusters as requested. If necessary the clusters
returned are fragmented. However fragmentation can be controlled by the min-fragement
parameter. FindFreeCluster will return the first cluster of the allocated cluster 
list. The FAT will be updated.

###AppendCluster(uint32_t clusterlist, uint32_t cluster)

not implemented

This will append a cluster to the last cluster of a cluster list, the FAT will be 
updated

###CreateDirEntry(struct* direntry, char* name)

not implemented

This will create a new directory entry.
